* cl-llama-cpp

Common Lisp bindings to [[https://github.com/ggml-org/llama.cpp][llama.cpp]] via CFFI.

** Status

*Active Development/Beta*
The low-level bindings cover the full public C API of llama.cpp and the bundled ggml and gguf libraries (=llama_=, =ggml_=, and =gguf_= symbols --- auto-generated via claw). A higher-level Lisp interface is provided for common inference workflows, while some advanced features currently require using the raw =%llama= bindings directly. These are the current high-level wrappers:

- Model loading and context creation
- Tokenization / detokenization
- Text generation with sampling (temperature, top-k, top-p, min-p, greedy, and extended samplers)
  - Extended samplers: typical-p, XTC, top-n-sigma, mirostat v1/v2, repetition/frequency/presence penalties, DRY anti-repetition, logit bias, dynamic temperature, adaptive-p
  - Individual sampler constructors (=make-temp-sampler=, =make-top-k-sampler=, =make-dist-sampler=, etc.) for building custom chains with full ordering control, without raw =%llama:= calls; =free-sampler= as the public destructor
  - Reusable sampler config objects (=make-sampler-config=) — bundle any sampler parameters into a plist; pass as =:sampler-config= to =generate= or =with-sampler-chain=; explicit keyword arguments always take priority over the config
- Token streaming via callback
- Chat template formatting and safe multi-turn tokenization
- Embeddings
- LoRA adapters
- KV cache management (clear, copy, shift, divide, query)
- Incremental chat session with KV-cache reuse (=make-chat-session=, =chat-session-send=, =chat-session-reset=) --- per-turn prefill cost proportional to new tokens only; common-prefix tracking avoids re-decoding cached history; graceful context-window recovery
- Session state save/load (file-based and in-memory persistence)
- Model & context introspection (description, metadata, properties, system info)
- GGUF file inspection (metadata key/value access, tensor index, version/alignment/offsets; =with-gguf= lifecycle macro)
- Grammar / constrained generation (GBNF grammars, lazy grammars)
- Infill generation
- Batch API and parallel decoding
- Backend lifecycle management (=with-backend=, =ensure-backend=, NUMA init, thread-safe init/free)
- Context runtime configuration (thread counts, mode flags, abort callbacks, threadpool management)
- Performance counters (structured timing data, =with-perf= macro)
- Logging (route llama.cpp log messages to a Lisp callback; thread-safe)
- System capability queries (mmap, mlock, GPU offload, RPC, device count)
- Backend device & registry introspection (device enumeration, names, types, VRAM, properties; registry enumeration)
- Resource planning & configuration validation (memory estimation, feasibility reports, configuration suggestions, pre-creation guardrails)
- Pre-flight input validation (type checks, bounds/range checks, encoding validation — clean Lisp conditions instead of segfaults)
- GC finalizers as a safety net for model/context handles (=trivial-garbage=; standalone =make-model=/=make-context= constructors with =free-model=/=free-context= destructors)

*Not yet wrapped* at the high level (use the =%llama= bindings directly --- see Low-level =%llama= Access):

- Model quantization
- Speculative decoding
- Multimodal inputs (vision, audio)
- Fine-grained sequence control

The generated =%llama= layer covers 100% of the public C API surface (881 functions, 38 enums, 59 structs), so a missing high-level wrapper is never a blocker. See the escape hatch documentation for naming conventions and usage.

The API is unstable and will change without notice.

** Beyond llama.cpp

These features provide functionality above what the llama.cpp C API offers directly:

- =generate-parallel= --- Complete multiple prompts simultaneously in shared forward passes. A single function call handles batch allocation, multi-sequence decoding, per-sequence sampling, and result collection. The C API provides batch primitives but no built-in parallel generation loop.

- =estimate-memory= / =feasibility-report= / =suggest-configuration= --- Estimate model + KV-cache + compute memory from GGUF metadata and validate a configuration against a VRAM budget before loading. Free VRAM is auto-detected from the GPU when no budget is supplied. Pure Lisp-side computation; llama.cpp has no equivalent planning API.

- *Pre-flight input validation* --- Every high-level wrapper validates its inputs (types, bounds, ranges, non-emptiness) at the Lisp layer before making the foreign call. Invalid inputs signal structured =input-validation-error= conditions instead of causing segfaults or undefined behavior in C. llama.cpp's C API performs minimal input checking and relies on the caller to pass valid data.

** Requirements

- *SBCL* (Recommended --- llama.cpp triggers floating-point exceptions that must be masked. This is handled automatically on SBCL; other implementations are supported via the configurable **llama-fp-wrapper**, though they may require implementation-specific trap masking to prevent crashes.
- [[https://github.com/ggml-org/llama.cpp][llama.cpp]] built as a shared library (=libllama.so= / =libllama.dylib=)
- [[https://cffi.common-lisp.dev/][CFFI]] and [[https://github.com/cffi/cffi][cffi-libffi]]
- [[https://github.com/trivial-garbage/trivial-garbage][trivial-garbage]]

*Platform notes:* Tested on Linux. macOS should work (the library loader handles =.dylib=). Windows is untested and would require building llama.cpp as a DLL.

** Installation

*** Building llama.cpp

The repository includes llama.cpp as a git submodule.

#+begin_src bash
git clone --recursive https://github.com/licjon/cl-llama-cpp
cd cl-llama-cpp/llama.cpp
cmake -B build -DBUILD_SHARED_LIBS=ON
cmake --build build -j$(nproc)
#+end_src

The library loader will automatically find =libllama.so= in =llama.cpp/build/bin/=. Alternatively, install it somewhere on your system library path.

*** Loading with ASDF

Place or symlink this directory into your ASDF source registry (e.g. =~/common-lisp/=), then:

#+begin_src lisp
(asdf:load-system "cl-llama-cpp")
#+end_src

For Roswell users, clone into =~/.roswell/local-projects/= and it will be found automatically.

If you use Quicklisp and want =ql:quickload= to find it, run =ql:register-local-projects= first.

** Quick Start

#+begin_src lisp
(asdf:load-system "cl-llama-cpp")

;; One-shot generation
(cl-llama-cpp:with-model (model "/path/to/model.gguf" :n-gpu-layers 99)
  (cl-llama-cpp:with-context (ctx model :n-ctx 2048)
    (princ (cl-llama-cpp:generate ctx "The capital of France is"
                                  :max-tokens 64
                                  :temp 0.7))))
#+end_src

*** Multi-turn Chat (Simple)

=examples/simple-chat.lisp= is the minimal reference. Every turn re-tokenizes
and re-processes the full history from scratch (O(n²) total work), so prefill
latency grows with conversation length. Fine for short demos.

#+begin_src lisp
(asdf:load-system "cl-llama-cpp/examples")

(cl-llama-cpp/examples/simple-chat:main
  :model-path "/path/to/chat-model.gguf")
;; User> What is Common Lisp?
;; AI> ... (reply, then prompts again)
;; User> Tell me more about CLOS.
;; AI> ... (full history re-prefilled each turn)
;; User> exit
#+end_src

*** Multi-turn Chat (Incremental / Fast)

=examples/incremental-chat.lisp= drives the same loop through =make-chat-session= +
=chat-session-send=. Each turn decodes only the new tokens; the KV cache is
reused for the already-decoded prefix. Prefill latency stays constant regardless
of history length. When the context window fills, the session resets
automatically with a user-facing message.

#+begin_src lisp
(asdf:load-system "cl-llama-cpp/examples")

(cl-llama-cpp/examples/incremental-chat:main
  :model-path "/path/to/chat-model.gguf"
  :n-gpu-layers 99)
;; User> What is Common Lisp?
;; AI> ... (only new tokens decoded each turn)
;; User> Tell me more about CLOS.
;; AI> ... (prefix reused from cache, O(1) prefill)
;; User> exit
#+end_src

For REPL use, call =chat-session-send= directly — the REPL itself is the
interactive loop (=main= blocks the REPL thread).

** API Reference

*Nullability convention:* Every high-level wrapper translates between Lisp =NIL= and C null pointers automatically. Pass =NIL= for optional pointer arguments (e.g. =set-abort-callback= fn, =attach-threadpool= batch threadpool); receive =NIL= for nullable returns (e.g. =model-chat-template= when no template exists). Allocation failures (null pointer from =llama_model_load_from_file=, etc.) signal a condition rather than returning =NIL=. Users never need to call =cffi:null-pointer= or =cffi:null-pointer-p=.

*Boolean convention:* Parameters that are semantically boolean in the C API (=:embeddings=, =:vocab-only=, =:offload-kqv=, =:no-perf=, etc.) accept both idiomatic Lisp booleans (=T= / =NIL=) and raw C integers (=1= / =0=). Integers pass through unchanged for backward compatibility; =T= becomes =1= and =NIL= becomes =0=. Other types signal a =type-error=. This applies to all =:char=-typed boolean fields in =llama_model_params= and =llama_context_params= when passed as keyword overrides to =with-model= and =with-context=.

*** Handle Types

The high-level API represents resources as *typed handle structs* rather than raw C pointers:

| Handle type           | Predicate               | Pointer accessor              | Created by                                                     |
|-----------------------+-------------------------+-------------------------------+----------------------------------------------------------------|
| =llama-model=           | =llama-model-p=           | =llama-model-pointer=           | =with-model=, =make-model=                                         |
| =llama-context=         | =llama-context-p=         | =llama-context-pointer=         | =with-context=, =make-context=                                     |
| =llama-sampler=         | =llama-sampler-p=         | =llama-sampler-pointer=         | =with-sampler-chain=, =make-grammar-sampler=, etc.                 |
| =gguf-context=          | =gguf-context-p=          | =gguf-context-pointer=          | =with-gguf=                                                      |
| =ggml-backend-device=   | =ggml-backend-device-p=   | =ggml-backend-device-pointer=   | =backend-dev-get=, =backend-dev-by-name/type=, =backend-reg-dev-get= |
| =ggml-backend-registry= | =ggml-backend-registry-p= | =ggml-backend-registry-pointer= | =backend-reg-get=, =backend-reg-by-name=                           |

Backend device and registry handles are *borrowed references* — the GGML backend owns the underlying objects and no =free= is needed. The predicates replace =cffi:null-pointer-p= for handle variables. When you drop to =%llama:= directly, use the pointer accessor to get the raw C pointer — see Low-level =%llama= Access.

*** Resource Management

- =with-model (var path &rest params) &body body= --- Load a GGUF model, bind to =var= as a =llama-model= handle, free on exit. Keyword params override =llama_model_default_params= (e.g. =:n-gpu-layers 99=). Signals =model-load-error= on failure. Delegates to =make-model=/=free-model= internally.

- =with-context (var model &rest params) &body body= --- Create an inference context from =model=, bind to =var= as a =llama-context= handle, free on exit. Keyword params override =llama_context_default_params= (e.g. =:n-ctx 2048=). Signals =context-creation-error= on failure. The extra keywords =:validation= (=:off=, =:warn=, =:error=) and =:vram-budget= (bytes) run a pre-creation resource check (see Resource Planning & Validation): =:warn= signals =configuration-unsafe-warning= and proceeds, =:error= signals =configuration-unsafe-error= and aborts. Delegates to =make-context=/=free-context= internally.

- =make-model path &rest params= --- Standalone model constructor. Same parameters as =with-model=. Returns a =llama-model= handle with a GC finalizer registered via =trivial-garbage=: if the handle is garbage-collected without being freed, the finalizer calls =llama_model_free= as a safety net. The caller owns the handle and should eventually call =free-model=. Signals =model-load-error= on failure.

- =free-model model= --- Free the foreign memory held by =model= and cancel its GC finalizer. Idempotent — calling on an already-freed model is a no-op. Returns =NIL=.

- =make-context model &rest params= --- Standalone context constructor. Same parameters as =with-context=. Returns a =llama-context= handle with a GC finalizer. The caller owns the handle and should eventually call =free-context=. Signals =context-creation-error= on failure.

- =free-context ctx= --- Free the foreign memory held by =ctx=, cancel its GC finalizer, and clean up any registered abort callback. Idempotent. Returns =NIL=.

- =with-gguf (var path &key no-alloc) &body body= --- Open a GGUF file at =path=, bind the context to =var= as a =gguf-context= handle, execute =body=, free on exit (including non-local exit). =:no-alloc t= skips ggml tensor allocation, which is faster for metadata-only reads and avoids needing a loaded backend. Signals =gguf-load-error= on failure. Does not require the llama backend to be initialized.

- =with-sampler-chain (var &key temp top-k top-p min-p seed greedy model grammar grammar-lazy grammar-trigger-words infill typical-p xtc-probability xtc-threshold top-n-sigma mirostat mirostat-v2 mirostat-tau mirostat-eta repeat-penalty frequency-penalty presence-penalty penalty-last-n dry-multiplier dry-base dry-allowed-length dry-penalty-last-n dry-seq-breakers logit-bias dynamic-temp-range dynamic-temp-exponent adaptive-p adaptive-p-decay sampler-config) &body body= --- Allocate a sampler chain, bind to =var= as a =llama-sampler= handle, free on exit. With *no keyword arguments*, allocates an empty chain — populate it inside =body= using =sampler-chain-add=. With keyword arguments, pre-builds the chain with the same options as =generate= (see Extended Sampler Keywords). Grammar, =:mirostat=, =:dry-multiplier=, and =:logit-bias= keywords require =:model=. =:mirostat= and =:mirostat-v2= are mutually exclusive. Pass =:sampler-config= (a plist from =make-sampler-config=) to supply defaults; explicit keyword arguments always override the config.

*** Backend Lifecycle

- =ensure-backend= --- Ensure the llama backend is initialized. Thread-safe; safe to call from any thread. Establishes a permanent hold: the backend will not be freed when an enclosing =with-backend= scope exits. The recommended pattern for multi-threaded use is to call =ensure-backend= once from the main thread before spawning worker threads. Subsequent calls are no-ops.

- =with-backend (&key numa) &body body= --- Initialize the llama backend, execute =body=, then call =backend-free= on exit (including non-local exit). Nesting and concurrent calls from multiple threads are both safe: only the first call to find the backend uninitialized initializes it; the last active scope to exit frees it. If =ensure-backend= was called first, =with-backend= will not free the backend on exit. When =:numa= is a GGML NUMA strategy keyword (=:distribute=, =:isolate=, =:numactl=, =:mirror=), calls =llama-numa-init= before executing =body=.

- =set-n-threads ctx n-threads n-threads-batch= --- Set thread counts on =ctx= for single-token decoding (=n-threads=) and batch decoding (=n-threads-batch=). Both are required in one call. Takes effect on the next decode. Both arguments must be non-negative integers.

- =set-warmup ctx warmup-p= --- Enable or disable the warmup pass on =ctx=. Disabling after the first decode avoids redundant computation.

- =set-causal-attn ctx causal-attn-p= --- Enable or disable causal (autoregressive) attention on =ctx=. Disable for bidirectional attention models; takes effect on the next decode.

- =set-embeddings ctx embeddings-p= --- Set the embedding-output flag on =ctx=. Must match the mode the context was created with: enable only on contexts created with =:embeddings= non-nil, disable only on contexts created without it.

- =synchronize ctx= --- Block until all pending async compute on =ctx= is complete. High-level accessors (e.g. =embed=) call this automatically before reading results; use it explicitly in batched pipelines that need finer-grained control over when the barrier occurs.

- =set-abort-callback ctx fn= --- Register a Lisp function as the abort callback on =ctx=. =fn= is called with no arguments and should return =T= to abort the current decode or =NIL= to continue. Pass =NIL= to clear a previously registered callback. The callback runs on a ggml worker thread: lexical variables captured in a closure are visible there, but Lisp dynamic (special) bindings established on other threads are not. Errors in =fn= are caught, stored in =*last-abort-callback-error*=, and generation continues (callback returns =NIL=); they never propagate across the FFI boundary. Returns =NIL=.

- =get-abort-callback ctx= --- Return the Lisp abort callback registered on =ctx=, or =NIL= if none is set. Thread-safe.

- =*last-abort-callback-error*= --- The last error condition caught inside the abort-callback panic boundary, or =NIL=.

- =attach-threadpool ctx threadpool &optional threadpool-batch= --- Attach an external =ggml-threadpool= to =ctx=. =threadpool-batch= optionally specifies a separate threadpool for batch operations; when omitted =threadpool= is used for both. The caller retains ownership; =detach-threadpool= does not free it.

- =detach-threadpool ctx= --- Detach the threadpool from =ctx=. Does not free the threadpool.

*** Operations

- =tokenize model text &key add-special parse-special= --- Tokenize =text= into a vector of token integers. Signals =type-error= if =text= is not a string.

- =detokenize model tokens &key remove-special unparse-special= --- Convert a token vector back to a string. Signals =type-error= if =tokens= is not a vector.

- =generate ctx prompt &key max-tokens temp top-k top-p min-p seed parse-special prompt-tokens token-callback grammar grammar-root typical-p xtc-probability xtc-threshold top-n-sigma mirostat mirostat-v2 mirostat-tau mirostat-eta repeat-penalty frequency-penalty presence-penalty penalty-last-n dry-multiplier dry-base dry-allowed-length dry-penalty-last-n dry-seq-breakers logit-bias dynamic-temp-range dynamic-temp-exponent adaptive-p adaptive-p-decay sampler sampler-config reset-context= --- Generate text continuing =prompt=. Returns three values: the generated string, a stop reason (=:eog=, =:length=, or =:callback=), and a =(simple-array fixnum (*))= of the sampled token ids (useful for callers that need to track exact KV-cache contents without a lossy re-tokenize round-trip). When =:reset-context= is =NIL=, the KV cache is not cleared before decoding — the caller is responsible for ensuring the cache is already in a consistent state (e.g. the current cached prefix matches the new prompt). The default (=T=) preserves the original behaviour: clear the cache and re-prefill the full prompt each call. Signals =type-error= if =max-tokens= is not a positive integer or =prompt= is not a string or vector (when =prompt-tokens= is not provided). When =prompt-tokens= is provided, uses the pre-tokenized input directly. When =token-callback= is provided, calls it with each decoded token string as it is produced; return =NIL= from the callback to stop generation early. When =grammar= is provided, constrains output to match the GBNF grammar string (=grammar-root= defaults to ="root"=). When =sampler= (a =llama-sampler= handle, typically from =with-sampler-chain=) is provided, =generate= borrows the chain without freeing it — the caller owns the lifetime. A non-blocking warning is issued if other sampler-building keywords are also supplied. When =:sampler-config= (a plist from =make-sampler-config=) is provided without a pre-built =:sampler= chain, its values supply defaults for any sampler parameter not given explicitly in this call — explicit keywords always take priority over the config. See the extended sampler keywords below for advanced sampling control.

- =generate-parallel ctx prompts &key max-tokens temp top-k top-p min-p seed parse-special= /...all extended sampler keywords.../ --- Generate text for multiple =prompts= in parallel using the batch API. Each element of =prompts= is a string or a pre-tokenized token vector. All sequences share the same sampler configuration; each gets a unique seed derived from =seed= + sequence-index. The context must be created with =:n-seq-max= >= =(length prompts)=. Returns two values: a list of generated strings and a list of stop reasons (=:eog= or =:length=).

- =embed ctx text &key normalize= --- Compute embeddings for =text=. Returns a vector of single-floats. Signals =type-error= if =text= is not a string; signals =input-validation-error= if =text= is empty. The context must be created with =:embeddings t= (or =1=). Automatically synchronizes pending async compute before reading from the GPU to ensure correct results.

*** Chat Templates

- =format-chat model messages &key template add-assistant-prefix= --- Format =messages= (a list of =(:role "user" :content "...")= plists) into a prompt string using the model's chat template. Signals =input-validation-error= if =messages= is empty or any message is missing a =:role= or =:content= string. Automatically maps the ="assistant"= role for models that expect ="model"= (e.g. Gemma).

- =tokenize-chat model messages &key template add-assistant-prefix= --- Like =format-chat=, but returns a token vector with template markers parsed as special tokens and message content parsed as regular text. This prevents model-generated text that resembles special tokens from corrupting subsequent turns. Use with =generate='s =:prompt-tokens= parameter.

- =model-chat-template model &optional name= --- Return the chat template string embedded in the model.

- =list-chat-templates= --- Return a list of built-in chat template names.

*** LoRA Adapters

- =with-lora (var model path) &body body= --- Load a LoRA adapter GGUF from =path= for =model=, bind to =var=, free on exit. Signals =lora-load-error= on failure.

- =apply-lora ctx adapter &key scale= --- Set the active LoRA adapter on =ctx=. =scale= (default 1.0) controls the adapter strength. Replaces any previously applied adapter. Signals =lora-apply-error= on failure.

- =lora-metadata adapter= --- Return metadata from =adapter= as an alist of =(key . value)= string pairs.

*** Extended Sampler Keywords

These keywords are accepted by =generate=, =with-sampler-chain=, and =build-sampler-chain=. They compose in a fixed order: logit bias → penalties → DRY → grammar → sampling strategy. All three also accept =:sampler-config= (a plist from =make-sampler-config=) as a source of defaults; explicit keyword arguments always win — see Sampler Config below.

*Token filtering:*

| Keyword           | Default | Description                                                |
|-------------------+---------+------------------------------------------------------------|
| =:typical-p=        |         | Keep tokens whose information content is near the expected |
| =:xtc-probability=  |         | Probability of excluding top choices (XTC)                 |
| =:xtc-threshold=    |     0.5 | Minimum probability for a token to be excludable           |
| =:top-n-sigma=      |         | Keep tokens within N standard deviations of the top logit  |
| =:adaptive-p=       |         | Dynamic top-p target probability (experimental)            |
| =:adaptive-p-decay= |     0.0 | Rate at which adaptive-p converges to target               |

*Repetition control:*

| Keyword             | Default | Description                                               |
|---------------------+---------+-----------------------------------------------------------|
| =:repeat-penalty=     |         | Penalise exact token repeats (1.0 = no penalty)           |
| =:frequency-penalty=  |         | Penalty scaling with token frequency (0.0 = off)          |
| =:presence-penalty=   |         | Flat penalty if token appeared at all (0.0 = off)         |
| =:penalty-last-n=     |      64 | How many recent tokens to consider for penalties          |
| =:dry-multiplier=     |         | DRY sequence-repetition penalty strength (requires model) |
| =:dry-base=           |    1.75 | DRY exponential base                                      |
| =:dry-allowed-length= |       2 | Minimum sequence length before DRY applies                |
| =:dry-penalty-last-n= |      -1 | Token window for DRY (-1 = use context size)              |
| =:dry-seq-breakers=   |         | List of strings that reset DRY matching (e.g. newlines)   |
| =:logit-bias=         |         | Alist of =(token-id . bias)= pairs (requires model)         |

*Sampling strategy:*

| Keyword                | Default | Description                                               |
|------------------------+---------+-----------------------------------------------------------|
| =:mirostat=              |         | Enable Mirostat V1 (requires model; replaces normal path) |
| =:mirostat-v2=           |         | Enable Mirostat V2 (replaces normal path)                 |
| =:mirostat-tau=          |     5.0 | Target surprise level (lower = more focused)              |
| =:mirostat-eta=          |     0.1 | Learning rate for mirostat adaptation                     |
| =:dynamic-temp-range=    |         | +/- range for entropy-adaptive temperature                |
| =:dynamic-temp-exponent= |     1.0 | Exponent shaping the dynamic temperature curve            |

*** Sampler Config

- =make-sampler-config &key temp top-k top-p min-p seed greedy grammar grammar-root grammar-lazy grammar-trigger-words grammar-trigger-patterns grammar-trigger-tokens infill typical-p xtc-probability xtc-threshold top-n-sigma mirostat mirostat-v2 mirostat-tau mirostat-eta repeat-penalty frequency-penalty presence-penalty penalty-last-n dry-multiplier dry-base dry-allowed-length dry-penalty-last-n dry-seq-breakers logit-bias dynamic-temp-range dynamic-temp-exponent adaptive-p adaptive-p-decay= --- Bundle sampler parameters into a reusable plist. Returns a plist containing *only the keys explicitly provided*; absent parameters carry no default — they are simply not in the plist. Pass the result as =:sampler-config= to =generate= or =with-sampler-chain=: the config supplies defaults for any parameter not given at the call site, and explicit keyword arguments at the call site always take priority. Unknown keywords signal =PROGRAM-ERROR= (SBCL also warns at compile time). Does not accept =:model= — resource handles have independent lifetimes and belong at the call site. The returned value is a plain plist; =getf= works directly on it.

*** Sampler Utilities

- =sampler-chain-add chain sampler= --- Add =sampler= to =chain=. Both arguments may be typed =llama-sampler= handles or raw CFFI pointers. The chain takes ownership: freeing the chain frees all added samplers. Returns =NIL=.

- =sampler-seed sampler= --- Return the current RNG seed from =sampler= as an integer. Useful for logging the seed that produced a given output.

- =free-sampler sampler= --- Free a =llama-sampler= handle created by any =make-*-sampler= function. Do not call on samplers that have been added to a chain — the chain owns and frees those. Returns =NIL=.

*Individual sampler constructors* — each returns a typed =llama-sampler= handle. The caller owns it and must either pass it to =sampler-chain-add= (transferring ownership to the chain) or free it with =free-sampler=. These enable building chains with full control over sampler ordering, without any =%llama:= calls.

- =make-greedy-sampler= --- Always picks the highest-probability token.
- =make-dist-sampler &optional (seed 42)= --- Random sampling weighted by softmax probabilities.
- =make-top-k-sampler k= --- Restrict candidates to the =k= highest-probability tokens.
- =make-top-p-sampler p &optional (min-keep 1)= --- Keep tokens until cumulative probability >= =p= (nucleus sampling).
- =make-min-p-sampler p &optional (min-keep 1)= --- Remove tokens with probability < =p= × max-prob.
- =make-typical-sampler p &optional (min-keep 1)= --- Locally typical sampling with probability mass =p=.
- =make-temp-sampler temp= --- Scale logits by =temp= before softmax.
- =make-temp-ext-sampler temp delta &optional (exponent 1.0)= --- Dynamic temperature: base =temp=, range =delta=, curve =exponent=.
- =make-xtc-sampler probability threshold &optional (min-keep 1) (seed 42)= --- XTC: trim high-probability tokens above =threshold= with probability =probability= per step.
- =make-top-n-sigma-sampler sigma= --- Keep tokens within =sigma= standard deviations of the max logit.
- =make-mirostat-v2-sampler seed tau eta= --- Mirostat V2 targeting perplexity =tau= with learning rate =eta=.

*** Grammar / Constrained Generation

*** Incremental Chat Session

A stateful abstraction that tracks exactly which tokens are resident in the
KV cache and decodes only the delta each turn.  The session *borrows* a
=llama-context= (no foreign resource of its own) — the caller is responsible
for the context's lifetime.

- =make-chat-session ctx &key template system-prompt= --- Create a chat session
  backed by =ctx=. Clears the KV cache to start clean. =template= overrides the
  model's embedded chat template. =system-prompt=, when given, seeds =messages=
  with an initial =(:role "system" :content ...)= entry. Signals
  =input-validation-error= if =ctx= is not a =llama-context=.

- =chat-session-send session content &rest generate-keys= → =(values reply stop-reason)= ---
  Append a user turn and generate a reply. Only the new tokens since the last
  turn are decoded; the cached prefix is reused. Additional keyword arguments
  are forwarded to =generate= (e.g. =:max-tokens=, =:temp=, =:seed=, =:sampler=);
  =:prompt-tokens= and =:reset-context= are stripped — the session manages
  those internally. On =decode-error= (context window exhausted), rolls back the
  KV cache and =messages= list atomically to the pre-call state before
  re-signaling, so the session remains consistent. Signals
  =input-validation-error= if =content= is not a non-empty string.

- =chat-session-reset session &key keep-system= --- Clear the KV cache and reset
  the conversation. When =:keep-system t= and the first message has role
  ="system"=, that message is retained; the next turn re-seeds the cache from
  it. Returns =NIL=.

- =chat-session-p session= --- Return =T= if =session= is a =chat-session=.

- =chat-session-context session= --- Return the =llama-context= backing the session.

- =chat-session-model session= --- Return the =llama-model= (derived from the context at creation time).

- =chat-session-messages session= --- Return the current message list as a list of =(:role … :content …)= plists.

See =examples/incremental-chat.lisp= for an interactive demo.

*** Resource Planning & Validation

Estimate memory use from GGUF metadata and validate a configuration against a VRAM budget --- before loading a model or creating a context. All sizes are in bytes. Snapshot =detect-free-vram= before loading for pre-load planning; after loading, the returned value reflects remaining headroom only.

- =estimate-memory model &key n-ctx type-k type-v= --- Return a plist =(:model-size :kv-cache :compute :total)= estimating memory for =model=. =n-ctx= defaults to the model's training context length; =type-k= / =type-v= default to =:f16= (the KV-cache quantization types, e.g. =:q8-0=). The =:compute= term is a fixed heuristic, so =:total= is an approximation.

- =explain-memory-usage model &key n-ctx type-k type-v stream= --- Print a human-readable memory breakdown to =stream= (default =*standard-output*=). Returns =NIL=.

- =validate-configuration model &key n-ctx type-k type-v n-gpu-layers vram-budget= --- Return a plist =(:status :safe/:unsafe/:unknown :reason string)=. Without =:vram-budget=, free VRAM is auto-detected via =detect-free-vram=; status is =:unknown= only when no GPU is present. When =:n-gpu-layers= is supplied, only the proportional GPU share of model weights (plus KV cache and compute) is checked against the budget.

- =feasibility-report model &key n-ctx type-k type-v n-gpu-layers vram-budget stream= --- Print an estimate + validation report to =stream= and return the combined estimate/validation plist. When =:vram-budget= is omitted, prints a "Detected VRAM" header showing free/total and device name(s).

- =suggest-configuration model &key n-ctx n-gpu-layers vram-budget= --- Return a plist =(:n-ctx n :n-gpu-layers n)= that fits within =:vram-budget=, or =NIL= if none is found. Without =:vram-budget=, free VRAM is auto-detected; returns =NIL= on CPU-only builds. Reduces =n-gpu-layers= first, then halves =n-ctx=.

*Runtime guardrails:* =with-context= accepts =:validation= (=:off=, =:warn=, =:error=) and =:vram-budget= (bytes). =:warn= signals =configuration-unsafe-warning= (a =warning=) and proceeds; =:error= signals =configuration-unsafe-error= and aborts context creation. Without =:vram-budget=, free VRAM is auto-detected from the GPU.

*** Batch API

For most use cases, =generate-parallel= (above) is sufficient. These lower-level batch functions are available when you need direct control over token-level batching, such as custom decoding loops or embedding batches.

- =with-batch (var n-tokens &key n-embd n-seq-max) &body body= --- Allocate a batch with capacity for =n-tokens=, bind to =var=, free on exit. =n-embd= non-zero allocates embedding slots instead of token slots. =n-seq-max= is the maximum sequence IDs per token (default 1). Signals =batch-init-error= if =n-tokens= <= 0 or allocation fails.

- =batch-add-token batch token pos seq-ids &key logits= --- Add one token to =batch= at position =pos= for the given sequence(s). =seq-ids= is an integer or a list of integers. =logits= when true requests logit computation for this token. Signals =batch-overflow-error= if =batch= is at capacity. Signals =input-validation-error= if =token= or =pos= is negative.

- =batch-add-embedding batch embedding pos seq-ids &key logits= --- Add one embedding vector to =batch=. =embedding= must match the =n-embd= used when creating the batch. Signals =input-validation-error= if =pos= is negative or =embedding= length mismatches =n-embd=.

- =batch-add-sequence batch tokens seq-id &key start-pos logits= --- Add a vector of =tokens= with sequential positions. =logits= controls which tokens get logit computation: =:last= (default), =:all=, or =nil=. Signals =input-validation-error= if =tokens= is empty or =start-pos= is negative.

- =batch-clear batch= --- Reset =batch='s active token count to 0. Existing slot contents remain allocated and will be overwritten by subsequent add calls.

- =batch-token-count batch= --- Return the current number of tokens in =batch=.

- =batch-decode ctx batch= --- Decode =batch= using context =ctx=. Signals =decode-error= on failure.

- =batch-encode ctx batch= --- Encode =batch= using context =ctx=. Signals =decode-error= on failure.

*** Low-level =%llama= Access (Escape Hatch)

The =%llama= package is a *supported, first-class escape hatch*. It exposes the complete public C API of llama.cpp and the bundled ggml and gguf libraries as CFFI bindings --- 881 functions, 38 enums, and 59 structs auto-generated from the upstream headers via [[https://github.com/borodust/claw][CLAW]]. If a high-level wrapper doesn't exist yet for something llama.cpp can do, drop down to =%llama:= and call the C function directly. A missing wrapper is never a blocker.

The high-level =cl-llama-cpp= API is an ergonomic convenience layer built on top of =%llama=. Both layers are equally supported; =%llama= is not an implementation detail.

**** Extracting Raw Pointers from Handles

The high-level macros bind *typed handles* (=llama-model=, =llama-context=, =llama-sampler=), not raw CFFI pointers. When you need to call a =%llama:= function directly, extract the raw pointer first:

#+begin_src lisp
(cl-llama-cpp:with-model (model "/path/to/model.gguf")
  (cl-llama-cpp:with-context (ctx model :n-ctx 512)
    (cl-llama-cpp:with-llama-compatible-fp-environment ()
      (let ((ctx-ptr (cl-llama-cpp:llama-context-pointer ctx))
            (model-ptr (cl-llama-cpp:llama-model-pointer model)))
        ;; ctx-ptr / model-ptr are raw CFFI :pointer values for %llama: calls
        (%llama:decode ctx-ptr batch)))))
#+end_src

The accessors are:
- =llama-model-pointer model= → raw =%llama:model= pointer
- =llama-context-pointer ctx= → raw =%llama:context= pointer
- =llama-sampler-pointer sampler= → raw =%llama:sampler= pointer
- =ggml-backend-device-pointer device= → raw =%llama:ggml-backend-dev-t= pointer
- =ggml-backend-registry-pointer reg= → raw =%llama:ggml-backend-reg-t= pointer

To test whether a variable holds the right kind of handle, use the predicates =llama-model-p=, =llama-context-p=, =llama-sampler-p=, =ggml-backend-device-p=, =ggml-backend-registry-p= (replacing =cffi:null-pointer-p= on the old raw-pointer API).

| =ggml_mul_mat=     | =%llama:ggml-mul-mat=  | =ggml_= prefix kept, underscores become hyphens      |
| =gguf_get_key=     | =%llama:gguf-get-key=  | =gguf_= prefix kept, underscores become hyphens      |
| =GGML_TYPE_F16=    | =%llama:ggml-type-f16= | =GGML_= prefix kept for enum values                  |

Everything lives in one package (=%llama=) regardless of which C library it came from.

**** Structs and Enums

C structs and enums follow the same naming convention as functions:

- *Structs* are accessed as =(:struct %llama:NAME)= in CFFI type specifiers. Opaque types (=llama_model=, =llama_context=, =ggml_context=, etc.) are pointer-sized --- pass them as =:pointer=.

- *Enums* are Lisp keywords. =LLAMA_SPLIT_MODE_LAYER= becomes =:layer=, =GGML_TYPE_F16= becomes =:f16=. CLAW strips the common enum-name prefix from values. Use =(cffi:foreign-enum-value '%llama:split-mode :layer)= for the integer, or just pass the keyword where a CFFI function expects the enum type.

- *Bitfields* work like enums but multiple values can be OR'd together. CFFI handles the masking.

**** Floating-Point Environment

llama.cpp's SIMD code triggers floating-point exceptions (denormals, inexact results) that SBCL traps by default. All =%llama= calls that enter C code must run inside:

#+begin_src lisp
(cl-llama-cpp:with-llama-compatible-fp-environment ()
  (%llama:backend-init)
  ;; ... all your %llama calls ...
  )
#+end_src

The high-level API handles this automatically; you only need it when calling =%llama= directly. On non-SBCL implementations, this expands to the configured FP-trap masking wrapper (see the library source for details).

**** Discovering Symbols

- =src/bindings.lisp= is the generated binding source --- grep it, or use =M-.= from your editor.
- =(apropos "DECODE" '%llama)= from the REPL lists all symbols matching a substring.
- Every =%llama= function has a docstring containing its C prototype, e.g. =(describe #'%llama:decode)=.
- =src/binding-deps.lisp= lists the specific =%llama= symbols the high-level API depends on and provides =(cl-llama-cpp:check-binding-deps)= to verify they're all present after regenerating.

**** Example: Dropping Down to =%llama=

Model quantization has no high-level wrapper. You can do it with =%llama= directly:

#+begin_src lisp
(cl-llama-cpp:with-llama-compatible-fp-environment ()
  (%llama:backend-init)
  (unwind-protect
      (cffi:with-foreign-object (params '(:struct %llama:model-quantize-params))
        (cffi:convert-into-foreign-memory
         (%llama:model-quantize-default-params)
         '(:struct %llama:model-quantize-params) params)
        (setf (cffi:foreign-slot-value params
               '(:struct %llama:model-quantize-params) '%llama:ftype)
              :q4-k-m)
        (%llama:model-quantize "/path/to/f16.gguf"
                               "/path/to/q4km.gguf"
                               params))
    (%llama:backend-free)))
#+end_src

*** Custom Sampler Chains

=with-sampler-chain= with no keyword arguments allocates an empty chain. Use =sampler-chain-add= to populate it with typed =llama-sampler= handles from the =make-*-sampler= constructors. The chain controls sampler ordering — samplers apply in insertion order. Pass the chain to =generate= via =:sampler= to use it without building a new chain internally.

#+begin_src lisp
(cl-llama-cpp:with-model (model "/path/to/model.gguf" :n-gpu-layers 99)
  (cl-llama-cpp:with-context (ctx model :n-ctx 2048)
    (let ((gs (cl-llama-cpp:make-grammar-sampler model my-grammar)))
      (cl-llama-cpp:with-sampler-chain (chain)
        ;; All typed handles — chain takes ownership, frees all on scope exit.
        ;; Ordering is explicit: grammar first, then temperature, then distribution.
        (cl-llama-cpp:sampler-chain-add chain gs)
        (cl-llama-cpp:sampler-chain-add chain (cl-llama-cpp:make-temp-sampler 0.3))
        (cl-llama-cpp:sampler-chain-add chain (cl-llama-cpp:make-dist-sampler 42))
        ;; generate :sampler borrows the chain; it is not freed by generate.
        (cl-llama-cpp:generate ctx "Output a JSON object:"
                               :max-tokens 128 :sampler chain)))))
#+end_src

*** Tool Calling

The chat template already handles model-specific prompt formatting. Tool
calling only requires three things: tool schemas in the system message,
parsing the model's tool-call output, and feeding the results back as a
=tool= message. No new library dependencies are needed.

=examples/tool-calling.lisp= shows the complete cycle. The example
instructs the model via the system prompt to emit tool calls in a specific
XML format, making parsing deterministic regardless of model family.

#+begin_src lisp

  :model-path "/path/to/model.gguf"
  :user-query "What's the weather in Tokyo and what time is it in UTC?")
;; === Turn 1: model response ===
;; <tool_call>{"name": "get_weather", "arguments": {"location": "Tokyo, Japan"}}</tool_call>
;; <tool_call>{"name": "get_time", "arguments": {"timezone": "UTC"}}</tool_call>
;;
;; === Tool: get_weather {"location": "Tokyo, Japan"} ===
;; === Result: 72°F and sunny in Tokyo, Japan ===
;;
;; === Tool: get_time {"timezone": "UTC"} ===
;; === Result: 15:42 UTC ===
;;
;; === Turn 2: model response ===
;; The weather in Tokyo is 72°F and sunny. The time in UTC is 15:42.
#+end_src

The loop in =run-demo= continues until the model produces no more
=<tool_call>= blocks, so multiple tool calls in a single turn are all
dispatched before the next generate call. The key pieces in the example:

1. *Tool schemas* — a JSON array built with =format= (use jzon/yason in production)
2. *System message* — embeds the schemas and the expected output format
3. =tokenize-chat= + =generate= — unchanged from ordinary chat
4. *Parser* — =parse-all-tool-calls= extracts every =(name . args-json)= pair
5. *Dispatcher* — calls the corresponding Lisp function per tool call
6. *Tool result message* — appended with role ="tool"= before the next turn

See =examples/tool-calling.lisp= for the full code including the JSON
parser, a note on native chat-template tool calling (Llama-3.1, Qwen-2.5),
and guidance on adapting the output parser per model family.

*** Parallel Contexts on Multiple Threads

Independent contexts can run concurrently on separate Lisp threads. A single context must never be used from more than one thread at a time. Call =ensure-backend= once from the main thread before spawning workers; the backend lock guarantees that concurrent =with-context= calls inside worker threads will never race on backend init/free.

#+begin_src lisp
;; Call ensure-backend once on the main thread before spawning workers.
(cl-llama-cpp:ensure-backend)

(cl-llama-cpp:with-model (model "/path/to/model.gguf" :n-gpu-layers 99)
  ;; Models are read-only once loaded; sharing across threads is safe.
  (let* ((prompts '("The capital of France is"
                    "The speed of light is approximately"
                    "Water freezes at"))
         (results (make-array (length prompts) :initial-element nil))
         (threads (loop for p in prompts for i from 0
                        collect (let ((prompt p) (idx i))
                                  (sb-thread:make-thread
                                   (lambda ()
                                     ;; Each thread creates and owns its own context.
                                     (cl-llama-cpp:with-context (ctx model :n-ctx 256)
                                       (setf (aref results idx)
                                             (cl-llama-cpp:generate ctx prompt
                                                                    :max-tokens 32
                                                                    :temp 0.0))))
                                   :name "llama-worker")))))
    (mapc #'sb-thread:join-thread threads)
    (loop for p in prompts for r across results for i from 1
          do (format t "[~D] ~A~A~2%" i p r))))
#+end_src

This example uses =sb-thread= directly; adapt =make-thread= / =join-thread= to bordeaux-threads, lparallel, or any other library --- cl-llama-cpp imposes no threading dependency. See =examples/parallel-threads.lisp= for a complete runnable version.

*** Backend Lifecycle

#+begin_src lisp
;; with-backend replaces bare ensure-backend calls and guarantees
;; backend-free on exit, even on non-local transfer of control.
(cl-llama-cpp:with-backend ()
  (cl-llama-cpp:with-model (model "/path/to/model.gguf" :n-gpu-layers 0)
    (cl-llama-cpp:with-context (ctx model :n-ctx 512)

      ;; Adjust thread counts at runtime without recreating the context.
      (cl-llama-cpp:set-n-threads ctx 4 4)

      ;; Time-limited generation via abort callback.
      ;; Pass a plain Lisp function — no CFFI needed.
      ;; Lexical variables captured by a closure ARE visible from ggml
      ;; worker threads; Lisp dynamic bindings are NOT.
      (let ((abort-flag nil))
        (cl-llama-cpp:set-abort-callback ctx (lambda () abort-flag))
        ;; Arm via a timer; cancel it if generate returns early.
        (let ((timer (sb-ext:make-timer (lambda () (setf abort-flag t)))))
          (sb-ext:schedule-timer timer 2.0)
          (unwind-protect
              (handler-case
                  (cl-llama-cpp:generate ctx "Count from one:"
                                         :max-tokens 512 :temp 0.1)
                (cl-llama-cpp:decode-error () "aborted"))
            (sb-ext:unschedule-timer timer)))
        (cl-llama-cpp:set-abort-callback ctx nil)))))
#+end_src

See =examples/backend-lifecycle.lisp= for a full walkthrough covering thread tuning, mode toggles, synchronize, and time-limited generation.

*** Long-lived Handles (GC Finalizers)

=make-model= and =make-context= create handles that outlive a dynamic extent. A GC finalizer ensures the foreign memory is freed even if the handle escapes and is later garbage-collected. Prefer =with-model=/=with-context= when possible; use the standalone constructors for global models, caches, or server-style lifetimes.

#+begin_src lisp
;; Store a model globally for a server-style application
(defvar *model* (cl-llama-cpp:make-model "/path/to/model.gguf" :n-gpu-layers 99))

;; Create contexts on demand, free explicitly when done
(let ((ctx (cl-llama-cpp:make-context *model* :n-ctx 2048)))
  (unwind-protect
       (cl-llama-cpp:generate ctx "Hello" :max-tokens 32)
    (cl-llama-cpp:free-context ctx)))

;; Explicit cleanup — cancels the GC finalizer
(cl-llama-cpp:free-model *model*)

;; If you forget free-model, the finalizer frees the foreign memory
;; when the Lisp object is garbage-collected. This is a safety net,
;; not a substitute for explicit cleanup.
#+end_src

(cl-llama-cpp:with-backend ()
  ;; Quick system overview
  (let ((caps (cl-llama-cpp:system-capabilities)))
    (format t "Devices: ~D  Registries: ~D  GPU: ~A~%"
            (getf caps :n-backend-devs)
            (getf caps :n-backend-regs)
            (getf caps :has-gpu)))

  ;; Enumerate all devices
  (dotimes (i (cl-llama-cpp:backend-dev-count))
    (let ((dev (cl-llama-cpp:backend-dev-get i)))
      (multiple-value-bind (free total)
          (cl-llama-cpp:backend-dev-memory dev)
        (format t "[~D] ~A (~A)  ~,1F / ~,1F GiB free~%"
                i
                (cl-llama-cpp:backend-dev-name dev)
                (cl-llama-cpp:backend-dev-type dev)
                (/ free 1073741824.0)
                (/ total 1073741824.0)))))

  ;; GPU-only summary
  (let ((gpus (cl-llama-cpp:gpu-devices)))
    (format t "~D GPU device(s) found~%" (length gpus))
    (dolist (props gpus)
      (format t "  ~A: ~,1F GiB free / ~,1F GiB total~%"
              (getf props :name)
              (/ (getf props :memory-free) 1073741824.0)
              (/ (getf props :memory-total) 1073741824.0))))

  ;; Detect total free VRAM for resource planning
  (let ((vram (cl-llama-cpp:detect-free-vram)))
    (if vram
        (format t "Free VRAM: ~,1F GiB~%" (/ vram 1073741824.0))
        (format t "No GPU devices detected~%")))

  ;; Lookup a device or registry by name
  (let ((cpu (cl-llama-cpp:backend-dev-by-type :cpu)))
    (format t "CPU device: ~A~%" (cl-llama-cpp:backend-dev-name cpu)))

  ;; Enumerate registries
  (dotimes (i (cl-llama-cpp:backend-reg-count))
    (let ((reg (cl-llama-cpp:backend-reg-get i)))
      (format t "Registry ~S: ~D device(s)~%"
              (cl-llama-cpp:backend-reg-name reg)
              (cl-llama-cpp:backend-reg-dev-count reg)))))

#+begin_src lisp
;; Estimate memory and check a configuration against a VRAM budget.
;; :vocab-only t loads just the metadata — no weights — so this is fast.
(cl-llama-cpp:with-model (model "/path/to/model.gguf" :vocab-only t)
  ;; Human-readable breakdown
  (cl-llama-cpp:explain-memory-usage model :n-ctx 8192)

  ;; Programmatic check + suggestion against an 8 GiB budget
  (let ((budget (* 8 1024 1024 1024)))
    (format t "~S~%" (cl-llama-cpp:validate-configuration
                      model :n-ctx 32768 :vram-budget budget))
    (format t "Suggested: ~S~%" (cl-llama-cpp:suggest-configuration
                                 model :n-ctx 32768 :vram-budget budget))))

;; Pre-creation guardrail: refuse a context that won't fit the budget
(cl-llama-cpp:with-model (model "/path/to/model.gguf")
  (handler-case
      (cl-llama-cpp:with-context (ctx model :n-ctx 32768
                                            :validation :error
                                            :vram-budget (* 1024 1024 1024))
        ctx)
    (cl-llama-cpp:configuration-unsafe-error (e)
      (format t "Refused: ~A~%" e))))
